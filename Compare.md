## 技术架构差异

### 1. 解析器架构
- **dash**：基于 Yacc 的解析器（`parser.y` → `parser.c`），具有完整的 POSIX 语法
  - 来自词法分析器的令牌流（`input.c`）
  - 使用 `union node` 结构的解析树
  - 所有 shell 构造的语法动作
- **rs-dash**：手写递归下降解析器，语法有限
  - 解析器内的简单令牌扫描
  - 无适当解析树，直接执行
  - 语法覆盖有限

### 2. 执行模型
- **dash**：带完整作业控制的 Fork-exec 模型
  - 进程组和会话管理
  - 全程信号处理
  - 后台/前台作业跟踪
  - 管道进程协调
- **rs-dash**：简化的进程生成
  - 基本 `std::process::Command` 使用
  - 无作业控制或进程组
  - 管道实现有限

### 3. 变量系统架构
- **dash**：带哈希表的复杂变量表（`vartab[VTABSIZE]`）
  - 变量属性：导出，只读，整数，数组
  - 函数的局部变量栈
  - 子shell的变量继承
  - 位置参数的特殊处理
- **rs-dash**：环境变量的简单 `HashMap<String, String>`
  - 无变量属性
  - 基本位置参数存储
  - 无局部变量作用域

### 4. 内置命令系统
- **dash**：表驱动调度（`builtins.def.in` → 生成代码）
  - 赋值内置命令的特殊处理（`-a` 标志）
  - 特殊内置命令（`-s` 标志）影响 shell 状态

  - 字符串分配优化

- **rs-dash**：带标准分配器的 Rust 所有权模型
  - 编译器安全的内存管理
  - 无自定义分配器优化
  - 不同的字符串管理方法

### 6. 错误处理
- **dash**：集中式错误处理（`error.c`）
  - `error()`, `warning()`, `syntaxerror()` 函数
  - 交互模式的错误恢复
  - 信号安全的错误报告
- **rs-dash**：带 `eprintln!()` 的基本错误打印
  - 有限的错误恢复
  - 无结构化错误类型

### 7. 扩展系统
- **dash**：复杂扩展管道（`expand.c`）
  - 多个扩展阶段：波浪号，参数，命令替换，算术，路径名

2. **性能风险**：早期分析，优化瓶颈
3. **兼容性风险**：使用现有脚本测试，匹配 dash 行为
4. **维护风险**：干净的架构，良好的文档
5. **跨平台风险**：两个平台上的持续测试

## 注释

- rs-dash 具有良好的基础，具有基本的 shell 功能
- Rust 实现提供了内存安全优势
- 跨平台支持是 rs-dash 的优势
- 许多缺失功能很复杂，但在 dash 源中有详细记录
- rs-dash 的模块化架构应有助于添加缺失功能
